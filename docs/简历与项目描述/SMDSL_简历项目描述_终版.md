# SMDSL (RoboIR) — 简历项目描述 · 终版

## 简历正文（可直接填入）

---

**项目名称**：SMDSL — 面向具身智能的空间约束中间语言与闭环验证系统

**技术栈**：Python / Gradio / DeepSeek API / LibreDWG / OpenCV

**项目背景**：视觉-语言-动作（VLA）模型采用端到端黑盒模式直接输出动作指令——当机器人靠近障碍物时，系统无法解释"为什么这里不能走"，发生碰撞后也无法追溯根因，在时序依赖的复杂任务中极易陷入局部最优。

**项目系统架构**：

系统采用四模块声明式管线架构。模块一是空间感知引擎，接收 CAD 图纸（DWG/SVG/JSON/PNG/OSM 五种格式），经工业 DWG 解析与形态学后处理后，生成连续的欧氏距离场以及安全代价 A* 拓扑地图。模块二是语义编译器，接收自然语言任务指令，通过 DeepSeek 大模型将指令编译为标准化的 STL 时空约束声明——大模型在此阶段只做逻辑翻译，禁止计算任何坐标、距离或角度数值，由五条硬性护栏防止空间幻觉。模块三是物理约束求解器，接收编译后的 STL 约束与用户指定的运动轨迹，在距离场上通过双线性子像素采样逐轨迹点计算鲁棒度指标 ρ = d_real − D_safe，当 ρ 小于零时判定为碰撞或违规，并精确输出违规点的世界坐标与碰撞深度。模块四是闭环自愈层，当物理求解器检测到违规时，自动生成结构化诊断报告（含违规坐标与定量碰撞深度），将该报告反馈给大模型执行最小责任层恢复——大模型仅修改违规的约束参数而不推翻整体任务意图，随后基于新的安全阈值通过 A* 自动重规划路径，并在三轮内完成再验证与修正。

**主要工作**：

从零独立设计并实现了整套面向机器人 2D/2.5D 避障规划的"空间约束编译-验证-自愈"闭环系统（约一万行 Python 代码，二十一个模块），作为将自然语言任务指令安全转化为可执行、可审计的机器人运动约束的概念验证沙盒。

空间感知引擎方面，实现了对五种 CAD 格式的统一派发解析管线。其中工业 DWG 图纸路径包含七层安全沙箱（扩展名白名单校验、文件大小上限控制、DWG 文件头魔数校验、CVE-2025-61154 R2004 版本特殊缓解、dwgread 工具可用性检测、subprocess 超时控制、输出编码回退链），以及面向工业图纸特有问题的三步形态学后处理——双线墙焊接解决工业 DWG 中两条平行线段表示墙体导致的"墙缝漏水"问题，外部泛洪剔除解决 PNG 白色背景和 SVG 被误判为自由空间的问题，薄墙桥接通过动态核尺寸闭运算打通被门框线分割的相邻房间。在 FloorplanQA 数据集的全部 1,981 个室内布局上完成批量测试，提取平均耗时 29.9 毫秒，零失败。

STL 约束编译与验证方面，设计了声明式中间语言 RoboIR 的 Schema 与编译器。编译器通过 System Prompt 嵌入五条硬性护栏（禁止大模型输出坐标/距离/角度数值、禁止使用时序命令词、禁止输出 Markdown 包裹、禁止引用环境外物体、强制输出声明式 STL 表达式），配合编译期枚举值校验和引用一致性检查，在编译阶段消灭大模型的空间幻觉。物理验证端通过双线性插值在距离场上实现子像素精度的轨迹点采样，越界点被统一处理为距离为零（越界即为碰撞），输出连续语义梯度 ρ 而非简单的二值碰撞判定。对四条典型轨迹（安全路径、贴墙擦边、穿越物体、走出边界）进行了定量对标验证，ρ 值分别为+0.150、−0.150（53个违规节点）、−0.300（26个违规节点）、−0.300（38个违规节点），语义与数值在代码、文档、输出 JSON 三处完全自洽。

LLM 闭环自愈方面，设计并实现了完整的"生成-验证-结构化反馈-修正-重验证"闭环链路。当约束被违反时，系统自动提取最深违规点（含时间戳、世界坐标、实际距离、鲁棒度偏差值），生成包含 FailureTaxonomy 九类错误分类的结构化诊断报告，将该报告作为大模型的修正上下文。大模型依据最小责任层原则仅调整被违反的约束参数而不改变任务意图，系统随后基于新的安全阈值调用 A* 重新规划绕行路径，在三轮内完成从碰撞到安全的完整自愈。

**项目状态**：代码仓库 github.com/Xuzc317/SMDSL_demo（feature/smdsl-upload 分支），含完整 README、四模块架构文档、子模块使用说明及 ISSUES_LOG（六个已记录的技术问题及其根因分析与修复建议）。

## 附录：项目实践中遇到的关键问题与解决过程（供面试准备，不写入简历）

以下记录了项目开发过程中真实遇到的技术难题、根因分析过程和解决方案。这些内容在面试中被追问"你具体做了什么"时可以直接引用。

### 问题一：工业 DWG 图纸"双线墙漏水"

**现象**：工业 CAD 图纸中墙体用两条平行线段分别绘制外墙皮和内墙皮，栅格化后两条线之间的像素被识别为"自由空间"。导致 EDT 距离场在墙体内部产生虚假的大距离值，A* 路径直接穿过墙体内部。

**定位过程**：对 historical_museum.dwg 运行管线后发现产出 87 个连通分量但仅 4 条边——这意味着大部分"区域"实际上是墙体内部的虚假缝隙。逐像素比对双线墙间距（10-20cm 物理墙厚），发现在 5cm/px 分辨率下缝隙约为 2-4px 宽。

**解决方案**：实现 weld_double_line_walls 函数。对障碍层做形态学闭运算——先膨胀（障碍向外扩张，两条平行墙线在中间相连）再腐蚀（障碍回缩，但被焊接的内腔永久变为障碍）。核半径由物理墙厚和分辨率动态计算：radius_px = ceil(max_gap_m / resolution / 2)，30cm 墙厚 @5cm/px → 3px 核半径。优先使用 OpenCV 的 MORPH_CLOSE（椭圆核，毫秒级），cv2 缺失时回退到 scipy.ndimage.binary_closing。

**效果**：连通分量从 87→4，区域碎片化问题根治。

### 问题二：外部自由空间泛洪

**现象**：PNG 图片的白色边框被二值化为自由空间（值=1），SVG 格式的 grid 初始化为全 1。EDT 在建筑外部也计算距离场，A* 漫水泄漏到建筑外部并错误地将建筑外圈识别为可通行路径。

**定位过程**：可视化 A* 访问热力图时发现建筑外围有大面积高访问频次区域。对比 grid 初始值逻辑——PNG 默认黑=墙/白=自由，但图片的白边恰好也是白色；SVG 初始化为全 1（自由），墙体线段画为 0，但建筑轮廓之外没有线段覆盖。

**解决方案**：实现 remove_exterior_freespace。用 scipy.ndimage.label 对自由空间做四连通分量分析，遍历栅格四边缘收集所有"外部连通域 ID"的集合，将外部连通域像素强制覆写为 0（障碍）。加入安全护栏：若覆写后内部自由像素低于原自由像素的 0.5%，说明输入本身不含封闭建筑边界（如线稿 PNG），保留原 grid 避免整张图被清零。

**效果**：PNG/SVG 路径下 A* 搜索严格限制在建筑内部，热力图不再泄漏到外部。

### 问题三：A* 对角移动"幽灵穿墙"

**现象**：A* 使用 8 邻域搜索时，对角方向移动可能在没有实际穿过门洞的情况下从两个正交墙体的墙角处"滑过"——此时对角线的两个正交邻居都是墙体，但 A* 没有检测这一条件。

**定位过程**：在厨房场景可视化中发现路径点出现在墙体的内角位置——该位置距离场值小于 R_robot，但 A* 仍将其标记为 CLASS_PATH。追溯到 A* 的邻居展开逻辑——对角移动仅检查了目标格子本身的距离场，未检查两个正交邻居。

**解决方案**：在 safety_aware_astar_flood 和 astar_shortest_path 中显式添加角切检测。对角线移动时（dx≠0 且 dy≠0），同时检查两个正交邻居的占据状态和距离场值——任一个为墙（grid=0）或距离小于 R_robot，则该对角移动被拒绝。这个检查需要同时应用于 Dijkstra 漫水（多源）和 A* 最短路径（点对点）两个函数。

**效果**：零幽灵穿墙，所有路径点严格满足物理防穿透约束。

### 问题四：大模型输出不稳定——DeepSeek API 的重试与容错

**现象**：DeepSeek 在 JSON 严格模式（response_format={"type": "json_object"}）下偶发返回不合法的 JSON——含 Markdown 代码块包裹、前缀解释文字、或服务端直接拒绝该参数。

**定位过程**：统计 API 调用日志发现约 8-12% 的调用返回非纯 JSON。三种失败模式：(1) ```json ... ``` 代码块包裹；(2) JSON 前面有解释文字；(3) DeepSeek 服务端对 json_object 模式的 TypeError。

**解决方案**：实现三级 JSON 解析容错链。第一级直接 json.loads（最快路径）→ 失败则正则剥离 ```json ... ``` 或 ```...``` 代码块 → 再失败则正则提取首个完整 {...} 对象。API 调用层对 response_format 参数做 try/except TypeError 兜底——如果服务端拒绝 json_object 模式，自动剥离该参数重试。整个 VlmParser 类具备 3 次重试 + 指数回退（1.5s × attempt）。

**效果**：API 调用成功率从约 88% 提升到接近 100%（3 次重试覆盖绝大多数瞬态失败）。

### 问题五：超大 DWG 文件——fish_processing_plant.dwg 的 320MB JSON 溢出

**现象**：鱼加工厂 DWG 含极高密度的工业设备实体，dwgread 输出 JSON 达到约 320MB、超过一千二百万行。dwgread 写入临时文件被操作系统 pipe buffer 截断，导致 json.loads 在约第三亿二千万字符处抛出 JSONDecodeError。

**定位过程**：通过 batch_summary.json 统计到 19/20 文件成功、1 个失败。检查失败文件发现其实体数量远超其他文件（hotel-ground-floor-plan 有 12.8 万实体，fish_processing_plant 更大）。单次 json.loads 解析 320MB 文本也接近 Python 内存舒适边界。

**解决方案**：当前阶段采取短期方案——在 parse_dwg_to_json 的 CVE-2025-61154 缓解逻辑基础上，将 MAX_DWG_SIZE_BYTES 从 50MB 收紧到适用于超大工业文件的合理上限，对超限文件返回明确的错误提示而非崩溃。ISSUES_LOG 中记录了长期方案：对大文件使用 ijson 流式解析或分块写入，或增加 LibreDWG 的实体数量限制。

**效果**：剩余 19 个文件全部成功解析，超大文件给出可理解的错误信息而非系统崩溃。

### 问题六：PNG 透明通道被错误二值化

**现象**：FloorplanCAD 数据集中的 PNG 图纸是"线稿+透明背景"格式——透明像素（alpha=0）被 PIL.Image.convert("L") 强制转为像素值 0（黑色），导致 99% 像素全黑，二值化后几乎不可用。

**定位过程**：对比原始 PNG 文件与二值化输出 grid——怀疑 convert("L") 对 alpha 通道的处理方式。验证：RGBA 图像直接转灰度时，PIL 将 alpha=0 的像素也参与灰度计算而非视为"无内容"。

**解决方案**：在 _parse_raster 中添加 RGBA/LA/P 模式检测——若图像含透明通道，先创建白色背景再 alpha-composite 将图像贴到白底上，然后灰度化。额外加了 Otsu 阈值兜底：若固定阈值导致 free_ratio 接近 0 或 1（全黑或全白），自动用 cv2.threshold(THRESH_OTSU) 重算阈值。

**效果**：透明背景 PNG 的线稿正确提取为墙体（黑色），无内容的透明区域保持为自由空间（白色）。
